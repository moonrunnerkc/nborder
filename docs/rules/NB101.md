# NB101: non-monotonic execution counts

`NB101` flags notebooks whose `execution_count` field decreases somewhere in source order. A cell with execution count `3` after a cell with execution count `5` means the notebook was not run top-to-bottom; the recorded execution path is something a reader cannot reproduce by clicking "Run All".

## Why it matters

Pimentel et al. (2019) found that 36% of notebooks on GitHub fail when re-run from a fresh kernel, and out-of-order execution is one of the largest contributors. The `execution_count` field is metadata that records the order Jupyter actually ran cells in, not the order they appear in the file. When those orders disagree, the result is a notebook whose outputs reflect a hidden state nobody can recreate.

This rule is a metadata-only check: it does not parse cell source, just walks `cell.execution_count` once. It catches the cheap-to-detect case where someone re-ran a cell in the middle of the notebook before pushing.

## Bad

```
Cell 0: execution_count=5  → first_value = 1
Cell 1: execution_count=3  → second_value = first_value + 1
Cell 2: execution_count=7  → third_value = second_value + 1
```

`3 < 5` indicates cell 1 was run before cell 0 in this kernel, but appears after cell 0 in source order. Anyone running `Restart and Run All` may get different results.

## Good

```
Cell 0: execution_count=1
Cell 1: execution_count=2
Cell 2: execution_count=3
```

Cells were run in source order, which is what a reader would reproduce.

## Auto-fix

`--fix=clear-counts` clears every code cell's `execution_count` to `null`. The aggressive variant (cell reordering) is reserved for `--fix=reorder`, which only triggers from cross-cell dataflow rules (NB201). Clearing counts is the safe default; it makes the notebook honest about not having a reliable execution order rather than asserting one.

## Configuration

No knobs. The rule has no false-positive surface that warrants tuning.
