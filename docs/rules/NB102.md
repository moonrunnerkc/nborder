# NB102: won't survive Restart-and-Run-All

`NB102` flags any name used in source order that no cell in the notebook defines. The notebook works in your live kernel because the name was set by something you ran earlier, deleted from the cell, and forgot about. On `Restart and Run All`, the import or assignment that defined the name is gone, and the notebook crashes.

## Why it matters

Quaranta et al. (2022) describe hidden-state bugs as one of the dominant pathologies in production notebook codebases. The kernel state outlives any individual cell, so deleted code keeps "working" for the author but breaks for everyone else. Static checking catches it before review without requiring a fresh-kernel rerun.

Unlike NB201 (where the definition exists but in the wrong cell), NB102 means there is no definition at all, anywhere in the notebook. There is no auto-fix; the human has to add the import, paste the value back in, or delete the use.

## Bad

```python
# Cell 0
print(df)
print(totally_undefined)
```

```python
# Cell 1
df = {"rows": 3}
```

`df` lands as NB201 (defined later in the same notebook). `totally_undefined` lands as NB102 because no cell defines it.

## Good

```python
# Cell 0
import pandas as pd
df = pd.DataFrame({"rows": [3]})
```

```python
# Cell 1
print(df)
```

Every used name has a defining cell that runs before it.

## Auto-fix

None. NB102 means the user lost code or never wrote it; there is nothing for nborder to insert that would not be a guess. The diagnostic includes the cell index so the user can jump to the failing cell directly.

## Configuration

No knobs.

## Related rules

- **NB201**: same diagnostic shape, but with a fix when the definition exists later in the notebook.
- **NB103**: more focused than NB102 for the specific case of stochastic libraries that should be seeded.
