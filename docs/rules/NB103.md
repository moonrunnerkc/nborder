# NB103: stochastic library used without seed

`NB103` flags the first use of a stochastic library API that runs before any seed call. Without a seed, the result of the run is irreproducible: rerunning the notebook gives different numbers, and any downstream metric (model accuracy, sample mean) drifts from one execution to the next.

## Why it matters

Reproducibility is a baseline requirement for sharing analytical work. The fix is mechanical (call the library's seed function with a fixed value) but easy to forget, and the resulting drift is invisible until a reviewer cannot reproduce the notebook's claimed results. nborder detects which stochastic library is actually imported and injects the modern API for that library when `--fix=seeds` runs.

The detection table:

| Imported library | Seed call injected |
|------------------|--------------------|
| `numpy` (`import numpy as np`) | `np.random.seed(42)` followed by `rng = np.random.default_rng(42)` |
| `random` (stdlib) | `random.seed(42)` |
| `torch` | `torch.manual_seed(42)` plus `torch.cuda.manual_seed_all(42)` if `torch.cuda` is also imported |
| `tensorflow` | `tf.random.set_seed(42)` |
| `jax` | Diagnostic only; JAX requires explicit `PRNGKey` plumbing through every random call |
| `sklearn` | Diagnostic only when an estimator is constructed with `random_state=None`; the fix is to pass a `random_state` value, not a global seed |

The numpy injection covers both APIs in two lines because the legacy `np.random.<func>` calls and the Generator-bound `rng.<func>` calls draw from independent state. `np.random.seed(SEED)` seeds the legacy global RandomState; `np.random.default_rng(SEED)` returns a seeded Generator the user can bind. Without both lines, a notebook that mixes the two APIs (or that the user later edits to use the legacy API) loses determinism, even though `nborder check` reports no diagnostics. Adopters using 0.1.0 through 0.1.3 should re-run `nborder check --fix` against any numpy notebook.

The seed cell respects the user's chosen alias when the library is already imported with a non-canonical name (`import numpy as numpy_lib` produces `numpy_lib.random.seed(42)` plus `rng = numpy_lib.random.default_rng(42)`).

## Bad

```python
# Cell 0
import numpy as np
values = np.random.rand(3)
```

`np.random.rand(3)` runs before any seed is set; the output drifts run-to-run.

## Good

```python
# Cell 0
import numpy as np
rng = np.random.default_rng(42)
values = rng.random(3)
```

Modern API, deterministic, no global RNG state mutated.

## Auto-fix

`--fix=seeds` injects a single seed cell at the top of the notebook covering every detected stochastic library. For numpy specifically the cell carries two lines so both the legacy global API and the modern Generator API are seeded:

```python
import numpy as np
np.random.seed(42)
rng = np.random.default_rng(42)
```

The cell respects the user's chosen alias and skips the import line when a parameters cell already imports the library. Running the fix a second time is a no-op because the injected `np.random.seed` call counts as "seeded" for legacy calls, and the rest of the registry behaves the same way.

The detector tracks legacy and Generator seeding for numpy independently. A notebook that calls `np.random.default_rng(SEED)` but then uses `np.random.rand(3)` still fires NB103 on the legacy call. The Generator-only suppression that 0.1.0-0.1.3 produced was a false green light.

JAX and `sklearn` cases are diagnostic-only: the fix would require threading a `PRNGKey` or constructing estimators with explicit `random_state`, both of which are user-driven decisions.

## Configuration

```toml
[tool.nborder.seeds]
value = 42
libraries = ["numpy", "torch", "tensorflow", "random", "jax", "sklearn"]
```

`value` sets the seed integer used in injected calls. `libraries` filters which entries from the registry are enabled; remove a library to skip both detection and fixing for it.

## Related rules

None. NB103 is the only rule in the NB1xx family that is specific to a single concern (stochasticity); the other rules are more general execution-order checks.
