# NB201: use-before-assign across cells

`NB201` flags a name used in cell `N` that is only defined in some later cell `M` with `M > N`. The kernel happily runs the use because the author had executed cell `M` before cell `N` in their interactive session; the source order disagrees with the actual data flow.

## Why it matters

Pimentel et al. (2019) is the canonical study showing roughly a third of notebooks on GitHub fail on a fresh-kernel rerun, and use-before-assign is the most common single cause. The fix is structural: cells need to be reordered so that every name is defined before its first use. nborder builds a cross-cell symbol dependency graph and runs a topological sort to do this automatically when the dependency graph is a DAG.

## Bad

```python
# Cell 0
print(df.head())
```

```python
# Cell 1
import pandas as pd
df = pd.DataFrame({"rows": [3]})
```

`df` is used in cell 0 but only defined in cell 1. Restart-and-Run-All crashes on cell 0.

## Good

```python
# Cell 0
import pandas as pd
df = pd.DataFrame({"rows": [3]})
```

```python
# Cell 1
print(df.head())
```

Source order matches data flow.

## Auto-fix

`--fix=reorder` topologically sorts the cells so that every dependency runs before its dependents. The fix bails when the dependency graph contains a cycle (cell `A` uses something from `B`, cell `B` uses something from `A`); a cycle cannot be resolved automatically and the diagnostic message tells the user which two symbols form the cycle so they can split them by hand.

When the reorder fix applies, all `execution_count` fields are cleared to `null` (NB101's safe default) so the rewritten notebook does not falsely advertise a stale execution order.

## Configuration

No knobs.
